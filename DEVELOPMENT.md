# Development Guide

## 文档目的

本文档用于约束该项目从“论文方案”走向“可运行工程”的开发方式，确保后续补代码时结构清晰、结果可复现、分析过程可解释。

## 开发原则

- 数据与结论必须可追溯。
- 分析代码、建模代码、优化代码要分层。
- 同一份原始数据不允许在多个脚本中重复硬编码清洗逻辑。
- 预测结果和采购决策结果必须可复现。
- 论文图表、分析结果、导出报表应统一从脚本生成。

## 推荐环境

- `Python 3.11+`
- 建议使用 `venv` 或 `conda`
- 建议使用 `Jupyter Notebook` 做探索，正式逻辑沉淀到 `src/`

## 推荐依赖

基础依赖建议如下：

- `pandas`
- `numpy`
- `matplotlib`
- `seaborn`
- `statsmodels`
- `scipy`
- `pulp`
- `jupyter`
- `pytest`

## 目录约定

```text
data/raw/           原始数据，只读
data/processed/     清洗后的数据
notebooks/          探索性分析
src/data/           数据读取、校验、清洗
src/features/       指标计算与特征分析
src/forecast/       时间序列预测模型
src/optimization/   采购优化模型
src/visualization/  图表与报告导出
outputs/figures/    图片输出
outputs/tables/     表格输出
outputs/reports/    结果摘要
```

## 编码规范

- 默认使用 `UTF-8` 编码。
- 文件和目录名优先使用小写英文与下划线。
- 每个模块只负责一类事情，不要把“读取数据 + 训练模型 + 画图 + 导出结果”堆进同一个文件。
- 函数应有明确输入输出，避免依赖隐式全局状态。
- 关键计算逻辑应保留简短注释，说明业务含义而不是解释语法。

## 数据规范

- 原始数据只放在 `data/raw/`，不得直接覆盖。
- 清洗后数据写入 `data/processed/`。
- 字段命名建议统一为英文，例如：
  - `date`
  - `fruit_name`
  - `market`
  - `price`
  - `unit`
- 价格单位必须统一，例如统一到 `元/kg`。
- 必须显式记录缺失值处理、异常值处理和重采样规则。

## 建模规范

### 特征分析

- 基础统计指标至少包括均值、标准差、最小值、最大值。
- 风险指标至少包括变异系数、最大回撤。
- 季节性分析应明确时间粒度，如日、周、月。

### 预测模型

- 每个水果品类单独建模。
- 至少保留一个传统基线模型，如移动平均或指数平滑。
- `ARIMA` 建模前需做平稳性检验和必要差分。
- 模型评估至少输出 `MAE`、`RMSE`、`MAPE` 中的两项。

### 决策优化

- 输入必须来源于预测结果与风险指标，不能手工改写。
- 目标函数应清楚区分成本项与风险项。
- 约束至少包括需求约束、库存约束、非负约束。
- 高波动水果建议增加单周期采购上限约束。

## Notebook 规范

- `notebooks/` 只做探索，不作为最终生产逻辑。
- Notebook 名称建议使用前缀编号，例如：
  - `01_data_cleaning.ipynb`
  - `02_volatility_analysis.ipynb`
  - `03_price_forecast.ipynb`
  - `04_procurement_optimization.ipynb`
- Notebook 中最终结论应沉淀回 `src/` 脚本。

## 输出规范

- 图表统一输出到 `outputs/figures/`
- 表格统一输出到 `outputs/tables/`
- 结论摘要统一输出到 `outputs/reports/`
- 输出文件名应带日期或版本号，便于论文引用和结果回溯

## 最小开发流程

建议每次改动遵循以下顺序：

1. 明确本次修改的数据范围和目标。
2. 更新或新增清洗逻辑。
3. 运行特征分析和模型训练。
4. 导出预测结果和采购方案。
5. 检查图表与表格输出是否一致。
6. 补充必要说明文档。

## 验证要求

最低验证要求如下：

- 数据清洗脚本可重复运行。
- 预测脚本能在样本数据上正常产出结果。
- 优化脚本能输出非空采购方案。
- 关键指标和导出文件路径明确。
